# Contributing to Efferva

We welcome contributions that advance the protocol's core mission: enabling sovereign Agent ownership and federated intelligence circulation.

---

## What We're Building

Efferva is a protocol project, not a platform or product company.

Contributions should align with these goals:
- **Sovereign Agent ownership**: every person controls their own Agent
- **Open compute access**: intelligence should not be enclosed by platforms
- **Federated intelligence**: collective wisdom without centralized control
- **Anonymous exchange**: public rules, private persons
- **Intent-based interfaces**: services discovered by capability, not brand

---

## What We're Not Building

Please do not submit contributions that:
- Turn this into a platform business
- Add token economics or speculative mechanisms
- Introduce unnecessary governance complexity
- Create vendor lock-in or platform dependency
- Use buzzwords without substance
- Prioritize marketing over protocol clarity

---

## Types of Contributions

### 1. Core Axioms
Found in `docs/` and `rfcs/`.

These define the philosophical and methodological foundation.

Changes here require careful consideration and community discussion.

### 2. Protocol Specifications
Found in `specs/`.

These are draft protocols. They evolve through RFC proposals.

Contributions should:
- Be precise and implementable
- Avoid premature optimization
- Prioritize user sovereignty
- Consider compatibility with existing systems

### 3. Reference Implementations
Found in `implementations/` (future).

These demonstrate how protocols work in practice.

Implementations should:
- Be minimal and clear
- Serve as teaching tools
- Not become the only way to implement the protocol

### 4. Examples
Found in `examples/` (future).

These show narrow-domain applications.

Examples should:
- Solve real problems
- Be simple enough to understand quickly
- Demonstrate protocol value without hype

---

## How to Contribute

### Issues
Open an issue to:
- Report bugs in documentation or specs
- Propose new protocol features
- Ask clarifying questions
- Suggest improvements

### RFCs
For substantial changes, submit an RFC:
1. Copy `rfcs/0000-template.md` (if it exists) or follow existing RFC format
2. Write a clear proposal with motivation, design, and alternatives
3. Submit as a pull request
4. Engage in discussion

### Pull Requests
For direct contributions:
1. Fork the repository
2. Create a feature branch
3. Make your changes
4. Write clear commit messages
5. Submit a pull request with context

---

## Style Guidelines

### Documentation
- Use clear, direct language
- Avoid jargon and buzzwords
- Write for long-term clarity
- Prefer short sentences
- Use definitions over slogans

### Code (future)
- Prioritize readability
- Write minimal, focused implementations
- Include comments only where logic is non-obvious
- Follow language-specific conventions

---

## Review Process

1. Maintainers review for alignment with core principles
2. Community provides feedback
3. Changes are discussed openly
4. Consensus is sought, not forced
5. Merged contributions become part of the protocol's evolution
