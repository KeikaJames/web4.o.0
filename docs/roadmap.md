# Web4.0 路线图

**注意：这是一个克制的路线图。我们不承诺不现实的时间表。**

---

## Phase 0: Repository and Doctrine

**当前阶段**

目标：
- 建立仓库结构
- 写出核心哲学和方法论
- 明确设计原则
- 给出协议草案
- 建立 RFC 机制

产出：
- docs/ (宣言、方法论、原则、架构概览)
- specs/ (协议草案)
- rfcs/ (RFC 机制和初始 RFC)
- 仓库治理文档

---

## Phase 1: Core Specs

**目标：完成核心协议规范**

重点工作：
- 完善 Sovereign Agent Container 规范
- 完善 Intent Interface Protocol 规范
- 完善 Open Compute Interface 规范
- 完善 Federated Intelligence Layer 规范
- 完善 Compatibility Layer 规范

产出：
- 可实现的协议规范
- 协议测试用例
- 协议演进路径

---

## Phase 2: Reference Implementations

**目标：提供参考实现**

重点工作：
- SAC SDK (Sovereign Agent Container SDK)
- IIP parser and executor
- Local-first agent runtime
- Memory encryption utilities
- Compatibility gateways

产出：
- 可运行的参考实现
- 实现文档
- 开发者指南

原则：
- 参考实现是教学工具，不是唯一实现
- 保持最小化和清晰
- 不引入不必要的依赖

---

## Phase 3: Interoperability and Examples

**目标：验证协议互操作性**

重点工作：
- 多实现互操作测试
- 窄场景案例开发
- 兼容层测试

窄场景案例：
- Travel (旅行规划和预订)
- Local services (本地服务发现和调用)
- Research assistant (研究助手)
- Personal commerce (个人商务)
- Scheduling (日程管理)

产出：
- 互操作性测试报告
- 窄场景案例代码
- 用户体验反馈

---

## Phase 4: Field Testing

**目标：在真实场景中测试协议**

重点工作：
- 选择一个窄领域进行闭环测试
- 收集真实用户反馈
- 迭代协议和实现
- 识别未预见的问题

原则：
- 先做窄场景闭环
- 不追求大而全
- 重视真实反馈
- 快速迭代

---

## Phase 5: Ecosystem Growth

**目标：支持生态发展**

重点工作：
- 支持第三方实现
- 支持第三方服务接入
- 支持第三方工具开发
- 建立协议治理机制

原则：
- 保持协议公共性
- 不制造平台依赖
- 鼓励分叉和实验
- 持续追问"新贵从哪里长出来"

---

## 非目标

我们明确不追求：
- 快速用户增长
- 平台化运营
- 代币经济
- 复杂治理结构
- 大而全的功能覆盖

---

## 时间表

我们不给出具体时间表。

原因：
- 这是协议项目，不是产品项目
- 质量优先于速度
- 社区参与度不可预测
- 技术演进路径可能调整

---

## 当前优先级

Phase 0 完成后，优先：
1. 完善 SAC 规范
2. 开发 SAC 参考实现
3. 选择一个窄场景进行概念验证

---

**先做窄场景闭环，再谈大规模应用。**
