# Methodology: Replacement Over Patching

Efferva advances by replacing old protocol fulcrums, not by patching old systems.

---

## Problem

Old protocols concentrate control.
Patching them does not redistribute control.
Adding layers on top does not change the foundation.

Replacement does.

---

## Method

### 1. Identify Fulcrums

What does the old protocol do?
What problem does it solve?
What are its core functions?

Do not dismiss these functions. Understand them first.

### 2. Separate Necessity from Habit

Which functions are necessary?
Which are historical habits?

Example:
- Document transfer is necessary; client-server architecture is habit
- Delegated authorization is necessary; platform-as-identity-provider is habit
- Software distribution is necessary; single app store is habit

Necessary functions must be rewritten.
Habits can be replaced.

### 3. Rewrite with New Primitives

New primitives should:
- Move control closer to users
- Reduce intermediaries
- Increase composability

Example:
- Replace "API call" with "intent interface"
- Replace "platform account" with "sovereign Agent container"
- Replace "cloud API" with "open compute interface"
- Replace "centralized model service" with "federated intelligence layer"

### 4. Maintain Compatibility

Do not cut off the old world immediately.

New protocols should call old APIs, access old websites, use old payment rails, run on old infrastructure.

But old protocols are no longer sovereignty layers. They are resource layers.

### 5. Ask Where New Hierarchies Grow

Every replacement creates new power centers.

Keep asking:
- Does the new protocol re-create dependency?
- Does the new protocol re-enclose resources?
- Does the new protocol re-monopolize access?

If yes, replace again.

---

## Precision

Precision means:
- Identify every fulcrum of the old system
- Replace each with a new primitive
- Leave no unnecessary dependency
- Create no new black boxes

Not "governance detail accumulation."
Not "complex rule design."

---

## Containment

Containment means:
- Old protocols lose central position
- Old protocols become compatibility layers
- Old protocols cannot continue to enclose new intelligence
- Replacement cost decreases over time

Not "declare old protocols invalid."
Not "forbid old systems."

---

## Example: Sovereign Agent Replaces Platform Account

Old fulcrums:
- Users need identity, storage, service discovery, payment capability

Historical habits:
- Identity provided by platforms
- Storage on platform servers
- Services recommended by platforms
- Payment through platform wallets

New primitives:
- Identity held by users (passkey / hardware key)
- Storage in user-controlled containers (local-first + encrypted backup)
- Services discovered by capability (intent interface)
- Payment through open rails (no single platform dependency)

Compatibility:
- Sovereign Agents can call old platform APIs
- Sovereign Agents can log into old websites
- Sovereign Agents can use old payment methods

Result:
- Users no longer depend on platforms
- Platforms become replaceable service providers
- Old protocols contained in compatibility layer

---

Efferva advances by replacement.
